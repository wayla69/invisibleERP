# แผนการนำระบบขึ้นใช้งานจริงบน Alibaba Cloud (เน้นต้นทุนต่ำใน 3 ปีแรก)

> **สถานะ:** ฉบับร่าง v1.0 · **วันที่:** 2026-06-23 · **ผู้ดูแล:** Platform
> **ขอบเขต:** วางแผนการ deploy ระดับองค์กรของ Invisible ERP บน **Alibaba Cloud** โดย
> **ตัดเรื่องการเตรียม NASDAQ/การตรวจสอบหนัก (SOC 2/ISO) ออกไปก่อน** และ **เน้นต้นทุนให้ต่ำที่สุดใน 3 ปีแรก**
> ไฟล์งบประมาณประกอบ: [`alibaba-cloud-3yr-budget-th.xlsx`](alibaba-cloud-3yr-budget-th.xlsx)

---

## 1. สรุปสำหรับผู้บริหาร

- **เลือก Alibaba Cloud ภูมิภาคกรุงเทพฯ (Bangkok)** เหมาะกับธุรกิจในไทย: ความหน่วงต่ำ ข้อมูลอยู่ในประเทศ
  (ดีต่อ PDPA โดยไม่มีค่าใช้จ่ายเพิ่ม) และราคาถูกกว่า AWS ในเอเชีย
- **กลยุทธ์หลัก = เริ่มลีนก่อน แล้วค่อยขยาย** — ปีแรกใช้เครื่องเดียว (รัน api + web + Postgres รวมกัน)
  แล้วค่อยแยกฐานข้อมูลและเพิ่ม HA ในปีถัดไปตามการเติบโตจริง
- **ตัวลดต้นทุนที่สำคัญที่สุด** คือ **การซื้อแบบ Subscription จ่ายล่วงหน้า 3 ปี** (ลดราคาได้ลึกที่สุด) —
  ซึ่งตรงกับโจทย์ที่มองกรอบเวลา 3 ปีพอดี
- **ประมาณการต้นทุนโครงสร้างพื้นฐาน 3 ปี ≈ 12,750 USD (≈ ฿459,000)** ตามเส้นทางที่แนะนำ
  หรือต่ำสุด **≈ 3,750 USD (≈ ฿135,000)** ถ้าคงระดับลีนสุด (Tier 0) ตลอด 3 ปี
- เทียบกับแผน AWS แบบลีนที่เคยประเมินไว้ (~29,000–54,000 USD/3 ปี) ประหยัดได้ **ราว 65–80%**
- **ยังไม่รวมค่าแรง/ค่าพัฒนา (build effort)** ซึ่งไม่ขึ้นกับผู้ให้บริการ และสถาปัตยกรรมที่ลีนกว่าก็ทำให้
  งานติดตั้งน้อยลงด้วย

## 2. ทำไมต้อง Alibaba Cloud

| เหตุผล | รายละเอียด |
|---|---|
| **มีภูมิภาคในไทย** | Alibaba เปิดดาต้าเซ็นเตอร์ในไทย (กรุงเทพฯ) ตั้งแต่ปี 2022 — ความหน่วงต่ำสำหรับผู้ใช้ไทย และข้อมูลอยู่ในประเทศ |
| **ถูกกว่าในเอเชีย** | ค่า compute/ฐานข้อมูลโดยทั่วไปถูกกว่า AWS ในภูมิภาคนี้ |
| **ส่วนลด Subscription จ่ายล่วงหน้า** | แบบ 1 ปี และโดยเฉพาะ **3 ปี** ลดราคาได้ลึกมากเทียบกับแบบจ่ายตามใช้ (Pay-as-you-go) |
| **เครดิตผู้ใช้ใหม่/ฟรีเทียร์** | มักมีเครดิตเริ่มต้นช่วยลดค่าใช้จ่ายปีแรก |
| **สแตกพอร์ตได้** | ระบบเราเป็นคอนเทนเนอร์ + PostgreSQL + ที่เก็บไฟล์แบบ S3 จึงย้ายเข้า Alibaba ได้ตรงตัว |

> **หมายเหตุ PDPA:** การโฮสต์ในภูมิภาคกรุงเทพฯ ทำให้ข้อมูลอยู่ในประเทศ "ฟรี" ซึ่งช่วยเรื่อง PDPA
> แต่ **PDPA ยังเป็นกฎหมายไทยที่ต้องปฏิบัติเสมอ** แม้เราจะยังไม่ทำ SOC 2/ISO ในตอนนี้

## 3. การแมปบริการ (จากแผน AWS เดิม → Alibaba Cloud)

| สแตกของเรา (แผน AWS) | บริการเทียบเท่าของ Alibaba Cloud |
|---|---|
| ECS Fargate / EC2 | **ECS** (เครื่องเสมือน) — หรือ SAE (คอนเทนเนอร์แบบ serverless) / ACK (Kubernetes) ในภายหลัง |
| Aurora / RDS PostgreSQL | **ApsaraDB RDS for PostgreSQL** (หรือรัน Postgres เองบน ECS ในช่วงแรก) |
| S3 | **OSS** (Object Storage Service) |
| ALB + WAF | **SLB/ALB** + **WAF** ของ Alibaba |
| Secrets Manager + KMS | **KMS Secrets Manager** |
| CloudWatch / OTel | **CloudMonitor** + **SLS** (Simple Log Service) |
| Route 53 / ECR | **Alibaba DNS** / **Container Registry (ACR)** |

> ไฟล์ Dockerfile และ `docker-compose.yml` ที่มีอยู่แล้ว (`apps/api/Dockerfile`, `apps/web/Dockerfile`)
> นำมาใช้กับ Alibaba ได้ทันที ไม่ต้องแก้แอป

## 4. กลยุทธ์ต้นทุน: เริ่มลีน แล้วค่อยแยกทีหลัง

จุดที่ประหยัดได้มากที่สุดในกรอบ 3 ปี **ไม่ใช่ตัวผู้ให้บริการ แต่คือการ "ยังไม่จ่าย" ค่าฐานข้อมูลแบบ
Managed Multi-AZ ตั้งแต่วันแรก** เริ่มจากเครื่องเดียวที่รัน `docker-compose` (api + web + Postgres รวมกัน)
แล้วค่อยแยกฐานข้อมูลแบบ managed เมื่อปริมาณงาน/ความเสี่ยงสมเหตุสมผล — เป็นแนวทางที่ย้อนกลับได้และเข้ากับ
วินัย cutover แบบ strangler-fig ที่เรามีอยู่

### ระดับการใช้งาน (Tiers)

| Tier | ใช้อะไร | ค่าใช้จ่าย/เดือน (แบบ Subscription 3 ปี) |
|---|---|---|
| **0 — ลีนสุด** | ECS 1 เครื่อง (4 vCPU/16 GB) รัน api + web + **Postgres ที่จัดการเอง** ใน Docker; เก็บไฟล์/สำรองขึ้น OSS; snapshot รายวัน | **~$60–110/เดือน** |
| **1 — โปรดักชันลีน** | ECS แอป 1–2 เครื่อง + **ApsaraDB RDS PostgreSQL (HA)** + SLB + OSS | **~$180–350/เดือน** |
| **2 — เติบโต** | Tier 1 + read replica/HA + สแตค **Dedicated** สำหรับลูกค้ารายใหญ่รายแรก | **~$400–800/เดือน** (+$150–400 ต่อ dedicated 1 ราย) |

## 5. เส้นทางที่แนะนำ + ต้นทุนรวม 3 ปี (TCO)

ค่อย ๆ ขยายระดับตามการเติบโตของธุรกิจ แทนที่จะซื้อ HA ทั้งหมดตั้งแต่ต้น (ตัวเลขจากไฟล์งบประมาณประกอบ):

| ปี | การตั้งค่า | ~ต่อเดือน (USD) | รวมทั้งปี (USD) | รวมทั้งปี (THB) |
|---|---|---:|---:|---:|
| **ปีที่ 1** | Tier 0 ลีนสุด (Postgres จัดการเอง, สำรองขึ้น OSS อัตโนมัติ) | ~$104 | **~$1,251** | **~฿45,000** |
| **ปีที่ 2** | Tier 1 — แยก Postgres ไป RDS (HA) + เพิ่มเครื่องแอป + SLB | ~$223 | **~$2,670** | **~฿96,000** |
| **ปีที่ 3** | Tier 2 — เพิ่ม replica/HA + Dedicated 1 ราย | ~$736 | **~$8,832** | **~฿318,000** |
| | | | **รวม 3 ปี ≈ $12,753** | **≈ ฿459,000** |

> **ทางเลือกประหยัดสุด:** ถ้าคงระดับ Tier 0 ตลอด 3 ปี (เครื่องเดียว, Postgres จัดการเอง, มีวินัยสำรองข้อมูล)
> ต้นทุนรวม 3 ปีจะเหลือ **~$3,750 (≈ ฿135,000)** — ถือเป็น "พื้น" ของค่าใช้จ่าย

### เทียบกับ AWS
- แผน AWS แบบลีน: ~$800–1,500/เดือน → **~$29,000–54,000 ตลอด 3 ปี**
- เส้นทาง Alibaba ที่แนะนำ: **~$12,750 ตลอด 3 ปี** (ถูกกว่าราว **65–80%**)
- ส่วนต่างนี้ครึ่งหนึ่งมาจาก *สถาปัตยกรรม* (Postgres จัดการเอง vs Managed Multi-AZ) อีกครึ่งมาจาก
  *ผู้ให้บริการ + การจ่ายล่วงหน้า 3 ปี*

## 6. ตัวช่วยลดต้นทุน (เฉพาะ Alibaba Cloud)

1. **ซื้อแบบ Subscription จ่ายล่วงหน้า 3 ปี** สำหรับ ECS + RDS — ส่วนลดลึกที่สุด (ควรทดสอบขนาดบนแบบรายเดือนก่อนแล้วค่อยล็อก)
2. **ใช้อินสแตนซ์แบบ burstable (e/t-series)** สำหรับเครื่องแอป — ราคาถูกและเพียงพอกับโหลด ERP ทั่วไป
3. **จัดการ Postgres เองบน ECS ในปีที่ 1** แล้วค่อยย้ายไป RDS เมื่อต้องการความสะดวก/HA
4. **ตั้ง OSS lifecycle** ย้ายไฟล์สำรองเก่าไป Infrequent Access / Archive — ราคาแทบไม่มี
5. **ใช้เครดิตผู้ใช้ใหม่/ฟรีเทียร์** ช่วยปีแรก
6. **Resource savings plan** เมื่อเข้าสู่สภาวะคงที่

## 7. ข้อควรระวัง (Trade-offs ของการทำให้ลีน)

- **Tier 0 = ไม่มี HA** (เครื่องเดียว = จุดล้มเดียว) และ **ต้องดูแล backup/patch/restore ของ Postgres เอง**
  ยอมรับได้สำหรับเฟสประหยัดต้นทุน **เฉพาะเมื่อ** ตั้ง snapshot รายวันขึ้น OSS อัตโนมัติ **และทดสอบกู้คืนจริง**
  — อย่าข้ามการทดสอบกู้คืน เพราะ backup ที่ไม่เคยทดสอบ ไม่นับเป็น backup
- ตัวเลขทั้งหมด **ยังไม่รวมค่าแรง/ค่าพัฒนา** (build effort)
- หากในอนาคตจะกลับไปเส้นทาง NASDAQ/SOC 2 จะต้องเพิ่ม HA + เครื่องมือตรวจสอบในตอนนั้น —
  แต่แผนนี้ไม่ปิดกั้น เพียงแค่ค่อย ๆ เลื่อนระดับขึ้นไป Tier 1/2

---

## 8. ขั้นตอนถัดไป (เริ่มได้โดยไม่กระทบแอปที่รันอยู่)

1. **ยืนยันภูมิภาคกรุงเทพฯ** และตรวจสอบราคา/บริการที่ต้องใช้บน Alibaba Cloud Pricing Calculator ของภูมิภาคจริง
2. เปิดบัญชี Alibaba Cloud + รับเครดิตผู้ใช้ใหม่ ทดสอบขนาดด้วย ECS 1 เครื่องแบบรายเดือนก่อน
3. นำ `docker-compose.yml` ที่มีอยู่ขึ้นไปรันบน ECS (Tier 0) เป็นสภาพแวดล้อม staging คู่ขนาน —
   ไม่แตะระบบ Railway ที่รันอยู่จนกว่าจะพร้อมตัดสลับ
4. ตั้งระบบสำรองข้อมูล Postgres → OSS อัตโนมัติ + **ทดสอบกู้คืน**
5. เมื่อมั่นใจขนาดแล้ว ค่อยแปลงเป็น **Subscription 3 ปี** เพื่อรับส่วนลด
