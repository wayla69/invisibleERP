# 37b — Tax-Point Model: เอกสารขอความเห็น/ลงนามรับรองจากที่ปรึกษาภาษี

> **สำหรับ:** ที่ปรึกษาภาษี / ผู้สอบบัญชีภาษีอากร (CPTA) / ที่ปรึกษากฎหมายภาษี
> **จาก:** ทีมพัฒนา Invisible ERP V2 · **วันที่:** 2026-07-07 · **อ้างอิงแบบเต็ม:** `docs/37-tax-point-model-design.md`
>
> **วัตถุประสงค์.** ระบบกำลังจะเปลี่ยนจากการลงวันที่ VAT ตาม *วันที่ออกใบกำกับภาษี* ไปเป็นการลงตาม
> **จุดความรับผิดในการเสียภาษีมูลค่าเพิ่ม (tax point)** ตามประมวลรัษฎากร เพื่อให้ VAT ตกในงวด (ภ.พ.30) ที่ถูกต้อง
> ก่อนเปิดใช้ logic นี้ (เฟส 5.1b เป็นต้นไป) ขอให้ท่าน **สอบทานและลงนามรับรอง** ตารางกฎและการตัดสินใจเชิงระบบ
> ด้านล่าง (ปัจจุบันได้วาง *โครงสร้างข้อมูลแบบไม่เปลี่ยนพฤติกรรม* ไว้แล้ว — `tax_point_date` มีค่าเริ่มต้น = วันที่ออก
> ใบกำกับ จึงยังไม่กระทบรายงานใด ๆ จนกว่าจะเปิดใช้หลังท่านลงนาม)

## 1. ขอบเขตที่ขอลงนามในรอบนี้
- **VAT ขาย (output tax point)** สำหรับช่องทาง **POS / AR / อสังหาริมทรัพย์ (ขายผ่อน)**
- **ไม่รวม** ในรอบนี้: ภาษีหัก ณ ที่จ่าย (WHT), ภาษีธุรกิจเฉพาะ (SBT/ภ.ธ.), ภ.พ.36 (บริการนำเข้า), การส่งออก, ฝากขาย/ตัวแทน
  → แยกเป็นเฟสถัดไป (ขอความเห็นภายหลัง)

## 2. ตารางกฎที่ขอรับรอง (อ้างประมวลรัษฎากร)

| # | กรณี | มาตรา/คำสั่ง | วิธีที่ระบบจะใช้ (tax point) | ✅ รับรอง / ✏️ แก้ไข |
|---|------|-------------|------------------------------|:---:|
| 2.1 | **ขายสินค้าทั่วไป** | ม.78 (1) | เร็วที่สุดของ { **ส่งมอบ**, โอนกรรมสิทธิ์, ได้รับชำระราคา, ออกใบกำกับภาษี } | ☐ |
| 2.2 | **ให้บริการทั่วไป** | ม.78/1 (1) | เร็วที่สุดของ { **ได้รับชำระราคา**, ออกใบกำกับภาษี, ได้ใช้บริการ } | ☐ |
| 2.3 | **ขายผ่อนชำระ/เช่าซื้อ (กรรมสิทธิ์ยังไม่โอน)** — ใช้กับโมดูลอสังหาฯ | ม.78 (2) + ป.36/2536 | จุดรับผิด **รายงวด เมื่อถึงกำหนดชำระแต่ละงวด** และออกใบกำกับทุกงวด | ☐ |
| 2.4 | **เงินรับล่วงหน้า/มัดจำ** | ม.78(1)(ข)/78/1; ป.36/2536 | ถ้าเป็น **ค่าตอบแทน (ชำระราคา)** → จุดรับผิดเกิดเมื่อรับเงิน + ออกใบกำกับ; ถ้าเป็น **หลักประกันคืนได้ (ไม่ใช่ค่าตอบแทน)** → ไม่เกิด | ☐ |

**คำถามเชิงนโยบายสำหรับข้อ 2.4:** ในระบบเราจะให้ผู้ใช้ระบุ "ประเภทเงินรับล่วงหน้า" (ค่าตอบแทน / หลักประกัน) ตอนรับเงิน
เพื่อตัดสินการเกิดจุดรับผิด — วิธีจำแนกนี้ท่านเห็นชอบหรือไม่ หรือควรอิงเกณฑ์อื่น (เช่น เงื่อนไขในสัญญา)?

## 3. การตัดสินใจเชิงระบบที่ขอรับรอง

| # | ประเด็น | วิธีที่ระบบจะใช้ | ✅ / ✏️ |
|---|---------|------------------|:---:|
| 3.1 | **การจัดประเภทสินค้า/บริการ** | ทุกรายการสินค้ามีธง `supply_type` = สินค้า/บริการ (ค่าเริ่มต้น = สินค้า) ใช้เลือกกฎ 2.1 vs 2.2 | ☐ |
| 3.2 | **การลงงวดในรายงาน ภ.พ.30 / รายงานภาษีขาย** | จัดกลุ่มตาม **`tax_point_date`** (ไม่ใช่วันที่ออกใบกำกับ) | ☐ |
| 3.3 | **ธุรกรรมสกุลต่างประเทศ (เฟส 5.2)** | คำนวณ VAT เป็น **บาท ด้วยอัตราขายของธนาคารพาณิชย์ (BOT) ณ วันที่จุดรับผิด** | ☐ |
| 3.4 | **การย้อนงวด** | ถ้า `tax_point_date` ตกในงวดที่ปิด/ยื่นแล้ว ระบบจะ **แจ้งเตือนเป็นรายการผิดปกติ** (ไม่กลืนเงียบ) | ☐ |
| 3.5 | **ข้อมูลเดิม (grandfather)** | เอกสารเดิมทั้งหมด `tax_point_date` = วันที่ออกใบกำกับ → รายงานย้อนหลังไม่เปลี่ยน | ☐ |

## 4. ตัวอย่างประกอบ (ขอยืนยันตัวเลข)

| กรณี | ข้อมูล | จุดรับผิดที่ระบบจะใช้ | ถูกต้อง? |
|------|--------|----------------------|:---:|
| ก. ขายสินค้า ส่งของ 10 มิ.ย. ออกใบกำกับ 15 มิ.ย. | goods | **10 มิ.ย.** (ส่งมอบก่อน) → VAT งวด มิ.ย. | ☐ |
| ข. ขายสินค้า รับมัดจำ(ค่าตอบแทน) 1 มิ.ย. ส่งของ 18 มิ.ย. | goods | **1 มิ.ย.** (รับชำระก่อนส่งมอบ) | ☐ |
| ค. บริการ ออกใบแจ้งหนี้/ใบกำกับ 18 มิ.ย. รับเงิน 25 มิ.ย. | service | **18 มิ.ย.** (ออกใบกำกับก่อนรับเงิน) | ☐ |
| ง. บริการ รับเงิน 12 มิ.ย. ออกใบกำกับ 20 มิ.ย. | service | **12 มิ.ย.** (รับเงินก่อน) | ☐ |
| จ. ขายผ่อนอสังหาฯ งวดครบกำหนด 1 ก.ค. / 1 ส.ค. / 1 ก.ย. | installment | จุดรับผิด **1 ก.ค., 1 ส.ค., 1 ก.ย.** (รายงวด) | ☐ |

## 5. คำถาม/ข้อวินิจฉัยที่ต้องการเพิ่มเติม
1. ข้อ 2.4 — เกณฑ์จำแนก "มัดจำที่เป็นค่าตอบแทน" vs "หลักประกัน" ที่รัดกุมพอสำหรับการ audit?
2. ข้อ 3.3 — ยืนยันว่าใช้ **อัตราขาย BOT ณ วันจุดรับผิด** (ไม่ใช่วันออกใบกำกับ/วันชำระ) สำหรับแปลง VAT เป็นบาท?
3. กรณี **ใบลด/เพิ่มหนี้ (ม.86/10, 86/9)** — จุดรับผิดของการปรับปรุงใช้วันที่ออกใบลด/เพิ่มหนี้ ถูกต้องหรือไม่?
4. ธุรกรรมที่ **จุดรับผิด ≠ วันออกใบกำกับ** โดยชอบด้วยกฎหมาย — มีข้อกำหนดเรื่องระยะเวลาการออกใบกำกับที่ต้องระวังหรือไม่?

## 6. ผลของการลงนาม
- **ลงนามรับรอง §2–§4** → ทีมพัฒนาจะเปิดใช้เฟส **5.1b** (คำนวณ `tax_point_date` ตอนออกเอกสาร), **5.1c** (จัดงวด
  รายงานตามจุดรับผิด), **5.1e** (ขายผ่อนรายงวด) พร้อมชุดทดสอบ (ToE) และปรับปรุงเอกสารกระบวนการ/UAT ตามนโยบายของ repo
- **ก่อนลงนาม** ระบบจะคง `tax_point_date` = วันที่ออกใบกำกับ (ไม่มีผลต่อรายงานหรือการยื่นภาษี)
