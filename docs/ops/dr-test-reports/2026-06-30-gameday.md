# DR Game-day Report — 2026-06-30 (first executed)

**Control:** ITGC-OP-02 · **Type:** automated DR test (data-tier + app bring-up) · **Result: PASS ✅**
**Executed by:** CI workflow `DR Game-day` (`.github/workflows/dr-gameday.yml`) on a `postgres:16` service,
exercising the real ops scripts `tools/ops/pg-backup.sh` + `restore.sh`. Orchestrator:
`tools/cutover/src/dr-gameday.ts`. Evidence artifact: the run's `dr-gameday-report` upload.

## Scenario

Total loss of the primary database → recover from the latest backup and bring the application back to a
ready, authenticated state. RTO clock runs from the simulated disaster (database dropped) to a
ready + login-passing app on the recovered database.

| **Verdict** | **PASS ✅** |

- **RPO = 0 loss:** key tables reconciled exactly between the backup snapshot and the restored database
  (`tenants` 1, `users` 1, `accounts` 68, `journal_entries` 0).
- **RTO 2.6 s ≪ 30 min target.** Note this is a small seeded dataset on a single CI cluster; a real
  production restore scales with data volume and offsite-fetch time — re-measure with production-sized data.

## Scope & caveats

- This automated game-day covers the **data-tier recovery + app bring-up** — the measurable, highest-risk
  core. It does **not** exercise **region/provider failover or DNS cutover**; those remain the annual
  **tabletop / live game-day** items in `docs/ops/dr-bcp-plan.md` §6.
- Same-cluster drill: the cluster-level `app_user` role persists; a true fresh-cluster recovery
  re-establishes it (the re-grant step does so).
