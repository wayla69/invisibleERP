1. Executive summary

Invisible ERP V2 is a multi-tenant, Thai-localized ERP built on NestJS (Fastify), Drizzle ORM and PostgreSQL with row-level security (RLS), targeting SOX/ICFR documentation standards ahead of a NASDAQ listing. The application demonstrates strong security engineering: the classic OWASP-Top-10 pitfalls for an ERP of this size are, in the overwhelming majority, already closed — and closed thoughtfully, with fail-closed defaults, layered controls, and extensive in-code rationale.

We found no critical vulnerabilities and no unauthenticated cross-tenant data leak in the audited paths. We did find two directly-exploitable High-severity issues (an SSRF filter bypass, verified empirically; and an API-key privilege-escalation / Segregation-of-Duties bypass), two High-severity architectural/configuration weaknesses in the tenant-isolation model, and a set of Medium and Low items that are mostly hardening gaps and inconsistencies rather than open doors.

The four issues that matter most

  1. SSRF guard bypass (High, verified). The outbound-URL SSRF guard fails to detect IPv4-mapped IPv6 literals (e.g. https://[::ffff:169.254.169.254]/), so a tenant-registered webhook can reach the cloud-metadata endpoint, localhost, and internal RFC1918 hosts. The guard is otherwise excellent — this is a single missed encoding.
  2. API-key scope escalation & maker-checker bypass (High). POST /api/platform/api-keys stores whatever scopes the caller asks for without capping them to the caller's own permissions. An AccessAdmin (whose only power is user administration) can mint a key carrying a broad transacting permission set — and, because two API keys are two distinct principals, can create-with-key-A / approve-with-key-B to defeat every self-approval control within its tenant.
  3. RLS has no DB backstop outside the request transaction (High, architectural). RLS is enforced only inside the per-request SET LOCAL ROLE app_user transaction. Every @NoTx handler, SSE stream, raw-client path and background job runs on the base connection — which, under the typical managed-Postgres superuser role, bypasses RLS entirely and relies solely on hand-written tenant filters.
  4. Insecure-by-default tenancy mode (High, configuration). The default TENANCY_MODE=single-company grants every tenant Admin a global RLS bypass (sees all tenants). The boot-time safety check only warns unless STRICT_TENANCY_BOOT=1. Mitigated in production by disabled self-signup and god-reserved Admin grants — but the secure mode is not the default.

Bottom line. The foundations — authentication, session management, password/crypto handling, CSRF, secrets hygiene, audit logging, and CI/CD supply-chain controls — are excellent and require no structural change. The work to do is (a) fix the two concrete High bugs, (b) harden the tenant-isolation model so the database, not application code, is the last line of defense on every path, and (c) close a handful of consistency gaps (one missing self-approval check, a permissive web CSP, a fail-open CORS default). None of this requires a re-architecture.

2. Scope & methodology

This was a white-box review with full source access, combined with logical exploit reasoning and targeted local reproduction. No testing was performed against a live production environment. Coverage was organized across seven parallel workstreams plus a direct review of the security core:

  • Authentication & session — login, PIN login, MFA/TOTP, refresh-token rotation, JWT signing/verification, lockout, cookies.
  • Authorization — role/permission model, guards, IDOR, missing-guard sweep across all *.controller.ts, Segregation-of-Duties / maker-checker enforcement.
  • Multi-tenant isolation — RLS policies, the tenant transaction interceptor, GUC/role plumbing, god/platform-admin bypass, act-as-tenant.
  • Injection — full SQL-injection sweep of every raw-SQL and dynamic-identifier sink.
  • Web frontend — CSP & security headers, XSS sinks, client-side auth gating, secret exposure, token handling.
  • Outbound safety — SSRF, webhooks (inbound & outbound), API keys, SCIM, file uploads, PDF/image/XML processing.
  • Secrets / config / supply chain — env validation, secret defaults, logging/PII, dependency risk, GitHub Actions pipeline.

Findings are rated Critical High Medium Low by real-world exploitability and blast radius under the deployed configuration. "Configuration-dependent" High findings are architectural weaknesses whose exploitability hinges on a deployment setting.

4. High-severity findings

High H-1 · SSRF guard bypassed by IPv4-mapped IPv6 literal

apps/api/src/common/net-guard.ts:33–44 (isPrivateV6) — reached via assertPublicUrl / isPublicUrl
What we found. The SSRF guard is well-designed — it re-resolves DNS immediately before each send (DNS-rebinding defense), blocks RFC1918/loopback/link-local/CGNAT/multicast, and handles the dotted IPv4-mapped IPv6 form. But when a URL host is an IPv4-mapped IPv6 literal, the WHATWG URL parser normalizes it to hex form, which the guard's dotted-only regex does not match. Execution then falls through to a branch that evaluates the empty string and returns "public".
Verified reproduction. Reproduced locally against the exact guard logic:
https://[::ffff:169.254.169.254]/  → host: ::ffff:a9fe:a9fe  → isPrivate: FALSE  *** BYPASS ***
https://[::ffff:127.0.0.1]/        → host: ::ffff:7f00:1    → isPrivate: FALSE  *** BYPASS ***
https://[::ffff:10.0.0.5]/         → host: ::ffff:a00:5     → isPrivate: FALSE  *** BYPASS ***

The offending regex only matches the dotted form:

const mapped = ip.match(/^::ffff:(\d+\.\d+\.\d+\.\d+)$/);   // never matches ::ffff:a9fe:a9fe
const head = ip.split(':')[0];   // '' → parseInt('0',16) = 0 → no private range fires → returns false
Exploit. A tenant registers a webhook (POST /api/platform/webhooks) with url = "https://[::ffff:169.254.169.254]/latest/meta-data/iam/security-credentials/". It passes validation at register time and again in sendOnce, so on the next matching event the server issues an outbound request to the cloud-metadata service (credential theft on IMDSv1), or to any localhost/RFC1918 internal service. Node's dual-stack resolver connects the mapped address to its underlying IPv4. This defeats the guard's entire purpose for IP-literal targets, and every future consumer of assertPublicUrl inherits the hole. The DNS-hostname path is not affected (dns.lookup returns the dotted form).
Remediation. Do not trust the string form the URL serializer emits. After parsing, detect any IPv4-mapped/IPv4-compatible IPv6 address regardless of hex vs dotted (extract the low 32 bits and run the v4 check), or reject all non-global-unicast IPv6 by prefix. Add regression cases for both forms.
✓ Resolved — PR #525 (merged). isPrivateV6 now parses both the dotted and hex IPv4-mapped forms (reconstructing the v4 address from the hi/lo hextets) and fails closed on any other ::ffff: / 64:ff9b: / ::<v4> embedding. Verified 12/12 against the exact bypass vectors — all three literals above now return isPrivate: TRUE and are blocked.

High H-2 · API-key issuance does not cap scopes to the minter's permissions

apps/api/src/modules/platform/api-key.service.ts:25–38 (issue) · common/guards.ts:76–86 (scope→permission expansion)
What we found. issue() persists whatever scopes[] the caller submits — const scopes = (dto.scopes ?? []).join(',') — with no validation against the minter's own permission set (the DTO is z.array(z.string()).optional(), no enum, no cap). At authentication time the guard turns those scopes into effective permissions: '*'/'admin' → the full role-default permission set; otherwise each scope becomes a literal permission key.
Exploit — privilege escalation. The AccessAdmin role holds only the users permission and is designed as a non-transacting access-administration role (SoD rule R01 keeps users apart from any transactional duty). An AccessAdmin calls POST /api/platform/api-keys {scopes:['*']}, receives an ierp_… key whose principal carries a broad transacting permission set (ar, creditors, approvals, exec, order_mgt, procurement, …), and uses it to post journals, approve profile/workflow items, etc. — powers they do not hold interactively.
Exploit — maker-checker / SoD bypass. A key principal's username is apikey:<prefix>, distinct from the human's username. Every maker-checker guard compares usernames (requestedBy === user.username). So minting two keys yields two distinct principals: create a document with key A, approve it with key B — the self-approval check never trips. This is a complete maker-checker bypass within the tenant for any holder of users.
Mitigating factors (stated honestly). The guard forces the key's role to Sales and pins the key's own tenantId, so a key gets no Admin role and no RLS bypass — the escalation is confined to the minter's own tenant and cannot grant Admin. Minting requires the users permission, so an ordinary transacting role cannot do this; the realistic actor is AccessAdmin. Within that scope, however, this is a real, unmitigated control break — precisely the SoD boundary the product's compliance posture depends on.
Remediation. Intersect requested scopes with the minter's effective permissions at issue() (reject or trim anything they do not hold). Additionally, record the minting human on the key and resolve maker-checker identity to that human, so a key cannot act as a second, independent approver.
✓ Resolved — PR #526 (merged). The chosen fix is identity binding (the more robust half): api_keys gained a created_by column (migration 0287) set to the minting human at issue(), and the guard now adopts that human as the key's principal — so the two-keys / create-with-A-approve-with-B maker-checker bypass is closed (both keys resolve to the same human, self-approval trips). AccessAdmin retains its intended power to provision keys for others; the machine apikey:<prefix> identity is kept only for the public-API surface and audit traceability. (ext/worldclass harnesses updated to assert the minter identity.)

5. High (configuration-dependent) — tenant-isolation architecture

These two findings share a theme: the tenant-isolation model is correct and fail-closed inside the request transaction, but its guarantees weaken at the edges and depend on deployment settings that are not secure by default. Together they are the most consequential items in this report.

High* H-3 · No DB-level RLS backstop outside the request transaction

apps/api/src/database/database.module.ts (base pool role) · common/tenant-tx.interceptor.ts:113 · confirmed via 0002_rls.sql
What we found. RLS is real and FORCEd on every tenant_id table, and the per-request interceptor correctly does SET LOCAL ROLE app_user + sets the tenant/bypass GUCs, so a forgotten WHERE cannot leak inside a normal request. The problem is what runs outside that transaction. The application connects with the DATABASE_URL role and only assumes app_user inside the request tx. On the typical managed-Postgres deployment the base role is a superuser (or holds BYPASSRLS), for which FORCE RLS does not apply. Evidence: etax.service.ts performs a raw INSERT into an RLS+FORCE table with no GUC set and succeeds — which is only possible if the base connection bypasses RLS.
Consequence. Every @NoTx handler, every @Sse stream, every raw PG_CLIENT query, and every background runInTenantContext job runs on that RLS-bypassing base connection. That is a large surface — public diner/QR ordering, member OTP/LINE auth, the SSO callback, PSP/Stripe/messaging/channel-adapter webhooks, coalition till routes, and scheduled jobs. Each relies entirely on hand-written tenant filtering; there is no database safety net. A single missing or incorrect tenant_id predicate on any of these paths is a silent cross-tenant read or write. We did not find a confirmed leak — the audited handlers do scope explicitly — but the risk is systemic and latent rather than contained by design.
Remediation. Run the application's base connection as a dedicated non-superuser, non-BYPASSRLS owner role. Then FORCE RLS fail-closes these paths too: with app.tenant_id unset, tenant_id = NULL returns zero rows instead of everything. Keep the few intentionally-unscoped auth-global tables (e.g. login_attempts) as explicit, reviewed exceptions. This converts "trust every handler" into "fail-closed by default" — the single highest-leverage hardening in this report.
✓ Resolved — PR #532 (merged). A new boot-time RLS-backstop check (assertRlsBackstop) probes the live base role (is_superuser / rolbypassrls) and, by default in production, refuses to boot when that role bypasses RLS — turning the latent, systemic risk into a loud, fail-closed gate. Provisioning SQL for a dedicated non-superuser owner role is documented in docs/ops/tenancy-model.md §1bis. Operational follow-up (yours): the current deployment runs with the explicit opt-out ALLOW_RLS_BYPASS_BASE_ROLE=1 until that role is provisioned — remove it to make the database the true last line of defense (tracked as an ops task).

High* H-4 · Insecure-by-default tenancy mode; safety check is warn-only

apps/api/src/common/tenant-tx.interceptor.ts:52,95–96 · tenancy-boot-check.ts · env.validation.ts:57
What we found. The default TENANCY_MODE=single-company makes bypass = preAuth || isAdmin || … — i.e. any user with role Admin receives app.bypass_rls='on' and sees and writes every tenant's data. The boot-time isolation check only escalates from a warning to a refusal when STRICT_TENANCY_BOOT=1; by default it logs and boots.
Scenario. An operator provisions two companies on one database (the product's core model — the platform owner provisions many tenants) but leaves TENANCY_MODE at its default. Company A's Admin can now read and write Company B's ledger, invoices, payroll and customers via the normal Admin path — no token forgery required.
Mitigating factors. In production, public self-signup is unconditionally disabled and the Admin role grant is reserved to the platform owner (god), so an outsider cannot mint an Admin; and the boot check does warn. The residual risk is that the secure mode is not the default and the guard is advisory. Because new tenants already self-assign org_id at provisioning, multi-company is a viable default.
Remediation. Default to multi-company (org-scoped Admin bypass), or make refuse the default once more than one tenant exists on the database. Promote the isolation check from warn to fail-closed in production.
✓ Resolved — PR #532 (merged). The tenancy boot check is now fail-closed by default: assertTenancyBootSafe refuses to boot in production when more than one tenant exists under single-company mode (the old STRICT_TENANCY_BOOT warn-gate is removed — fail-closed is the default), with an explicit, documented opt-out (ALLOW_SINGLE_COMPANY_MULTI_TENANT) for a deliberately-accepted single-company deployment. Production topology is multi-company, so per-company org isolation holds. ToE in cutover/tenancy-boot.ts (21 checks) + unit tests.

6. Medium-severity findings

Medium M-1 · Web CSP allows script-src 'unsafe-inline' in production

apps/web/next.config.mjs:25

script-src 'self' 'unsafe-inline' ships in production. 'unsafe-inline' means any injected inline <script> or inline event handler executes — the load-bearing XSS defense is effectively opted out. The session JWT is httpOnly (so injected script cannot exfiltrate it directly — good), but the readable CSRF token is reachable by JS, so injected script can still perform authenticated state-changing actions via fetch. The code documents nonce-based hardening as a tracked follow-up. Fix: adopt Next.js nonce-based CSP and drop 'unsafe-inline' from script-src. ('unsafe-eval' is correctly dev-only; the rest of the CSP — object-src 'none', base-uri 'self', frame-ancestors 'none', restricted connect-src — is strong.)

✓ Resolved — PR #541 (merged). The CSP moved from the static header into a per-request Next.js middleware nonce: production script-src is now 'self' 'nonce-<rand>' 'strict-dynamic' 'unsafe-inline', so a CSP3 browser ignores 'unsafe-inline' and runs only the nonce'd Next bootstrap — an injected inline <script> / event-handler is refused. Legacy browsers fall back to today's policy (no regression); a CSP_REPORT_ONLY=1 switch allows a safe observe-first rollout. Verified in real Chromium: header nonce == script nonce per request, app hydrates with zero self-inflicted violations, injected <img onerror> blocked.

Medium M-2 · Server-side SSRF via tenant-configurable OIDC issuer

apps/api/src/modules/identity/identity-config.service.ts:66 (stores oidc_issuer unvalidated) → sso.service.ts:157 (exchangeCode fetch)

upsert() validates oidc_redirect_uri for an http(s) scheme but applies no validation to oidc_issuer — no scheme check, no assertPublicUrl. exchangeCode then does fetch(\`${oidcIssuer}/token\`, …). A tenant admin (permission users; and in default single-company mode every self-service signup mints an Admin) sets oidc_issuer to an internal host or http://169.254.169.254/…, enables SSO, and drives the public, unauthenticated GET /api/auth/sso/callback flow. The server POSTs to the attacker-chosen internal endpoint; res.status is reflected in the error, providing a status/timing oracle. Fix: validate oidc_issuer with assertPublicUrl (https-only) at write time and re-check before exchangeCode. (Note this depends on the same SSRF guard as H-1 — fix H-1 first.)

✓ Resolved — PR #537 (merged). Two-layer fix mirroring the net-guard's register-and-re-check pattern: write time (identity-config.upsert) requires a well-formed https:// issuer → 400 BAD_ISSUER (DNS-free, never false-fails offline); send time (exchangeCode) calls assertPublicUrl(tokenUrl, { allowHttp:false }) before the outbound POST → 400 SSRF_BLOCKED on any internal / metadata / RFC1918 / loopback target. Uses the H-1-hardened guard. ToE added to the ext harness (private-IP issuer via the auth-code path → SSRF_BLOCKED).

Medium M-3 · approveLeave has no self-approval (maker-checker) check

apps/api/src/modules/hcm/hcm.service.ts:104–118 (compare with approveTimesheet at :57–58)

The adjacent approveTimesheet correctly blocks ts.submittedBy === user.username with SOD_SELF_APPROVAL. approveLeave has no equivalent guard — it approves and bumps the paid-leave balance regardless of who created it. requestLeave sets createdBy: user.username, and both routes share the same class-level @Permissions('exec','users','creditors'), so one holder can request paid leave and approve their own. Lower blast radius than a financial control, but a genuine, inconsistent maker-checker gap on a payroll-affecting workflow. Fix: mirror approveTimesheet — reject when lr.createdBy === user.username.

✓ Resolved — PR #527 (merged). approveLeave now throws SOD_SELF_APPROVAL when lr.createdBy === user.username, exactly mirroring approveTimesheet. The hcm harness approves as a distinct user and asserts a self-approve attempt is rejected; the troubleshooting FAQ documents the code.

Medium M-4 · CORS silently defaults to localhost:3000 in production

apps/api/src/main.ts:41 · env.validation.ts:51–53 (warn-only)

const origins = (process.env.CORS_ORIGINS ?? 'http://localhost:3000')…, and the env validator only warns when CORS_ORIGINS is unset. With credentials: true, CORS correctness is security-relevant. A prod deploy that omits the variable boots allowing only localhost — primarily a self-DoS (the real web origin is blocked) rather than an exposure, but it is a fail-open default in a security control. Fix: promote CORS_ORIGINS to a required prod secret, or refuse the localhost fallback when NODE_ENV=production.

✓ Resolved — PR #528 (merged). CORS_ORIGINS is now a required production secret in the fail-closed env front door — a prod boot without it is refused rather than silently falling back to localhost. .env.example and the ops-observability test updated.

Medium M-5 · Front-door env validation gated entirely on NODE_ENV==='production'

apps/api/src/common/env.validation.ts:28–29

The required-secret assertions (DATABASE_URL, JWT_SECRET, APP_ENC_KEY, PSP_WEBHOOK_SECRET) run only when NODE_ENV is exactly production. A prod deploy where NODE_ENV is unset or misspelled skips the entire front-door gate. Compensating controls hold the line — auth.module.ts and crypto.ts independently throw unless NODE_ENV is development/test, so JWT and encryption keys still fail closed on an unknown env. The residual gap is PSP_WEBHOOK_SECRET/DATABASE_URL. Fix (belt-and-suspenders): treat any NODE_ENV other than development/test as prod-strict.

✓ Resolved — PR #536 (merged). validateEnv is now a no-op only for the two known-safe local values (development, test); every other NODE_ENV (unset, misspelled, staging, prod, Production, …) runs the full fail-closed required-secret gate — closing the residual DATABASE_URL/PSP_WEBHOOK_SECRET/CORS_ORIGINS gap. 10/10 env-validation tests (staging/prod/Production throw on missing secrets; a configured staging boots).

Medium M-6 · document.write of server HTML into a CSP-free popup

apps/web/src/app/(internal)/print/page.tsx:36 · apps/web/src/lib/terminal.ts:60

Receipt/print flows fetch server-rendered HTML and inject it via window.open()…document.write(html) (or into an about:blank iframe). Such a document inherits the app origin and, per M-1, an 'unsafe-inline' CSP — so any <script> in the HTML executes in the authenticated origin. Receipts render user-controlled fields (item/customer names, notes). We verified the primary server-side renderer (common/doc-html.ts, a4-template.ts) HTML-escapes all interpolated data via esc()/escAttr(), which neutralizes the chain today. This remains a Medium because the escaping is the only control on these popup paths (they have no CSP of their own) and any future receipt/invoice renderer that interpolates without esc() becomes stored XSS. Fix: render receipts via srcdoc with a tight per-frame sandbox (no allow-scripts) — receipts need no scripting — and keep the server escaping as a hard invariant.

✓ Resolved — PR #539 (merged). Both sinks now render the receipt into a sandboxed, script-free iframe via srcdoc (terminal.printHtmlSlip: sandbox="allow-same-origin allow-modals", no allow-scripts, so win.print() still works but scripts are inert; the print page's openReceipt uses a fully-sandboxed <iframe>). Verified in real Chromium: a hostile receipt (inline <script>, img onerror, svg script, svg/body onload) renders its text but every script vector stays inert, and print() remains callable. Server-side esc() escaping is retained as a hard invariant.

7. Low-severity & defense-in-depth

✓ All 12 low findings resolved & merged. Each is fixed exactly as recommended below; the per-item pull request is in the findings-register Status column. Highlights: L-2 shipped as an additive HMAC-over-body layer (the static secret keeps working until a partner opts into HMAC per-source — no integration breaks), and L-8 as an opt-in shared-Redis rate-limit store + Fastify trustProxy (single-node behaviour byte-identical), which also corrects the L-12 audit-IP source. L-3 rode the guard's existing per-request users read at zero added cost. New unit/harness assertions accompany each.
  • L-1 · Inbound PSP webhook lacks replay protection. HMAC-over-raw-body is verified (good), but there is no timestamp/nonce window, so a captured valid callback can be replayed to re-drive a money-moving status flip. Mitigated by out-of-band status re-verification and idempotent handling. Add an explicit |now − ts| > 300s reject, matching the outbound signer. (pos-terminal.controller.ts)
  • L-2 · Static shared-secret webhooks. channel-adapter, restaurant/channel-order, and email-capture authenticate with a static bearer secret (constant-time compared — good) but not an HMAC over the payload, with no replay binding. Mitigated by ext_event_id idempotency and tenant-scoped RLS writes. Upgrade to HMAC-over-body where the sender supports it.
  • L-3 · tenantId trusted from the JWT claim. The guard re-reads role and orgId live from the DB but takes tenantId from the token. A user administratively moved to another tenant keeps their old RLS scope until the token expires. Source tenantId from the same per-request user read for consistency. (guards.ts:156–161)
  • L-4 · Provisioning does not reject god usernames. isPlatformAdmin() grants global bypass on username membership in PLATFORM_ADMIN_USERNAMES alone. Provisioning/signup check only uniqueness — so if a god name is listed before its account exists, or the god row is ever hard-deleted while still listed, an ordinary provisioning path that claims that username yields an instantly-global-god account. Reject any admin_username ∈ platformAdminUsernames() at provisioning. (billing.service.ts)
  • L-5 · developer.service relies solely on RLS. portal() ignores its user arg and selects api_keys with no tenant predicate; setTier() updates by id with no tenant predicate. Blocked only by RLS on api_keys (gated by users). Add an explicit tenant filter, as api-key.service.revoke does.
  • L-6 · Product-image fetch has no SSRF guard. urlToDataUrl fetches without assertPublicUrl, but the URL comes from a fixed Wikimedia Commons upstream, not user input — low reach. Route through the guard for defense-in-depth. (procurement/image-fetch.service.ts)
  • L-7 · SSE null-tenant broadcast. bi.controller.ts fans a live event with tenant_id == null to every connected tenant. Current publishers set tenant_id, but the null-is-global default is easy to trip, and SSE has no DB backstop (see H-3). Make the SSE filter fail-closed on a missing tenant.
  • L-8 · Rate limiting is per-process/in-memory, and no trustProxy. The edge limiter and the public-API limiter keep per-process counters, so under horizontal scaling (or WEB_CONCURRENCY>1) the effective ceiling is max × replicas. Separately, no Fastify trustProxy is configured while the audit log reads X-Forwarded-For first — behind the platform proxy the per-IP throttle key and audit IP can be inaccurate or (for the audit IP) spoofable. Move throttle state to the shared Redis adapter already present, and set a correct trustProxy.
  • L-9 · Object-storage key not sanitized. ${base()}/${key} is built by concatenation. All current callers pass server-generated keys (no traversal today), but a future caller passing user input containing ../ or an absolute URL could redirect the PUT/DELETE. Add key validation now. (object-storage.ts)
  • L-10 · SSO nonce check conditional on presence. if (claims.nonce !== undefined && claims.nonce !== st.nonce) skips the binding when the id_token omits a nonce. Mitigated by single-use server-side state consumption (the true replay defense), but validate nonce unconditionally when it was requested. (sso.service.ts:100)
  • L-11 · PII redaction is Thai-format only. The LLM-boundary phone regex matches Thai shapes; international numbers embedded in free-text memos may reach the model. Acknowledged as a minimization layer, not the complete control (a signed DPA + pseudonymization is the full control). Broaden the pattern. (pii-redact.ts:19)
  • L-12 · Audit IP is spoofable. clientIp() takes the first X-Forwarded-For value without validating it against a trusted proxy, so a client can set the IP recorded in the tamper-evident audit trail. Derive the audit IP from a trusted-proxy configuration. (audit.interceptor.ts:126–131)

8. What is done well

These are not filler — they are the reason the finding count is as low as it is, and they should be preserved through any refactor.

Authentication & session

  • JWT algorithm pinned both directions (algorithm: 'HS256' on sign, algorithms: ['HS256'] on verify) — closes alg:none and RS/HS key-confusion.
  • No committed fallback secret anywhere; dev without JWT_SECRET mints an ephemeral per-process random key; prod fails closed at both the module and the env front door.
  • The auth guard re-reads the live DB for role, active/deactivated, tenant-suspended, tokens_valid_from watermark, must-change-password, and privileged-MFA on every request — a forged or stale token claim cannot escalate, and a hard-deleted user's token stops working immediately.
  • Refresh-token rotation with reuse detection (a replayed rotated/revoked token revokes the whole family); refresh cookie is httpOnly and path-scoped to /api/auth.
  • Per-account lockout (DB-backed, survives restarts) layered under per-IP and stricter per-auth/per-OTP edge buckets. TOTP MFA with AES-256-GCM-encrypted seeds; PIN login hard-blocked for any MFA-requiring role.

Cryptography & secrets

  • scrypt password hashing with self-describing, ratcheted work factors and transparent rehash; the legacy unsalted-SHA-256 path was removed (weak-hash sink eliminated). Constant-time comparisons throughout.
  • AES-256-GCM AEAD at rest with a versioned HKDF keyring (label-separated keys), fail-closed on missing/malformed key material.
  • Webhook verification is HMAC-SHA256 over the raw body with a constant-time, length-hiding comparison.
  • Layered fail-closed secret validation; no private keys, default passwords, or committed credentials (gitleaks with a tightly-scoped i18n allowlist confirms this).

Authorization & SoD

  • Every controller resolves to at least one authz decorator. Self-approval (maker-checker) is enforced consistently across ~40 flows — GL posting/reversal, PMR/project change-orders, progress billing, subcontracts, payments, petty cash, 3-way match, tax notes, timesheets, and the central workflow engine (which also closes the delegate-approval-to-self hole).
  • Role escalation is explicitly blocked: granting Admin is reserved to the platform owner; SoD-conflicting grants route through a two-person control that blocks self-approval and self-grant.
  • Portal/ESS/payroll :id handlers are ownership-scoped; HQ cross-tenant targeting is defense-in-depth guarded before it reaches services.

Data, platform & supply chain

  • DB-enforced, FORCEd RLS with transaction-local GUCs and a fail-closed refusal if app_user cannot be assumed in production (the real backstop inside a request).
  • Tamper-evident, hash-chained append-only audit log (each row binds the previous), with cross-tenant access explicitly flagged on mutations.
  • PII minimization at the LLM boundary plus a DPA fail-closed gate; the exception filter never leaks stack traces or internal messages to clients.
  • Well-built SSRF guard concept (DNS-rebind re-check before send), high-entropy hashed API/SCIM tokens with constant-time compare, strict upload MIME/size allow-lists, no XXE (XML is built-and-escaped, not parsed with external entities).
  • CI/CD: no pull_request_target (the classic secret-exfil vector is absent), least-privilege GITHUB_TOKEN, no script-injection sinks, and gitleaks + CodeQL + pnpm audit as hard gates; deploys are approval-gated in a protected environment.

9. Cleared during review (no action needed)

  • SQL injection — none found. A full sweep traced every raw-SQL and dynamic-identifier sink. All resolve to Drizzle/postgres.js bound parameters or sql.raw() identifiers sourced from compile-time allowlists (the date_trunc period map; the PII key-rotation table/column const list). No string concatenation into SQL, no dynamic ORDER BY/direction from request input, no sql.identifier() on user input. The one string-built-identifier pattern (bi.service pii-rotation) is safe only because its list is a hardcoded const — worth a guard comment against future change.
  • Web XSS sinks — clean. No dangerouslySetInnerHTML, innerHTML, eval, or new Function anywhere in apps/web/src (only the document.write print paths of M-6).
  • Client secret exposure — clean. Only non-secret NEXT_PUBLIC_* public config is in the bundle; the session JWT is httpOnly and never in localStorage; the SSO callback avoids the sensitive-GET-query sink by forwarding the raw query in the POST body.
  • Open redirect — none. The login bounce sets a ?next= param but the login page never reads it back.
  • alg-confusion in the SSO id_token verifier — closed. verifyHs256 pins alg === 'HS256' and rejects everything else, with a constant-time signature compare; SSO also uses single-use server-side state + PKCE and fails closed on an empty client secret.
