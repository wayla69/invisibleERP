# Process Narratives — Invisible ERP V2

**Status: DRAFT v0.1 · 2026-06-22**

This folder holds **ISO-style documented procedures (process narratives + workflows)** for every financially-significant business cycle of the Invisible ERP V2 application. They are the human-readable companion to the control matrices in `compliance/` and are written so that an external financial auditor / IT auditor can perform a **walkthrough** of each process and tie every control step to a Risk & Control Matrix (RCM) control ID.

> **Disclaimer:** These are engineering/control working drafts to accelerate SOX 404(a) / ISO 9001 audit readiness. They are not legal, accounting, or compliance advice. Validate scope, control selection, and testing with your independent audit firm and a qualified SOX advisor. All `<<placeholders>>` must be completed by the process owner before these documents are approved and made effective.

---

## 1. ISO framing — why these documents exist

These narratives are structured to satisfy the **ISO 9001:2015 process approach** and its **documented-information** requirements:

- **Clause 4.4 — Quality management system and its processes:** each cycle is documented as a process with defined inputs, outputs, sequence, interactions, criteria, responsibilities (RACI), and risks/opportunities.
- **Clause 7.5 — Documented information:** each narrative carries a document-control header (ID, owner, approver, version, effective date, revision history) and is version-controlled in this repository.
- **Clause 8 — Operation:** the process narrative + flowchart describe operational control of each cycle, including controls of nonconforming outputs (Section 13, Exception & error handling).

The same documents double as **SOX/COSO process narratives**: each control step references an RCM control ID so that Test of Design (ToD) and Test of Operating Effectiveness (ToE) can be performed against documented procedure.

---

## 2. How to read each narrative

Every cycle file follows an identical 14-section structure:

1. Document control header · 2. Purpose · 3. Scope · 4. References · 5. Definitions & abbreviations
6. Roles & responsibilities (RACI) · 7. Process narrative (numbered, with endpoints + inline control IDs) · 8. Process flow (Mermaid flowchart + swimlane note) · 9. Control matrix
10. Inputs & outputs · 11. Records & retention · 12. KPIs/metrics · 13. Exception & error handling · 14. Revision history

## 3. Legend — RACI convention

| Code | Meaning |
|---|---|
| **R** | Responsible — performs the activity |
| **A** | Accountable — owns the outcome; single accountable role per activity |
| **C** | Consulted — provides input (two-way) |
| **I** | Informed — kept up to date (one-way) |

Roles in the RACI tables are the application's **single-duty roles** (least-privilege design): Cashier, PosSupervisor, ArClerk, ApClerk, Buyer, WarehouseOperator, InventoryController, StockCounter, GlAccountant, FinancialController, MasterDataAdmin, PricingManager, CreditManager, ReturnsClerk, AccessAdmin, ExecutiveViewer — plus the coarse roles Admin, Sales, Customer, Warehouse, Procurement, Planner where relevant. Segregation of duties (SoD) is shown by ensuring the **maker** role and the **checker/approver** role for any approval are never the same role, consistent with SoD rules R01–R16.

## 4. Legend — control-type convention

| Dimension | Values |
|---|---|
| **Purpose** | **Preventive (Prev)** — stops the error/fraud before it occurs · **Detective (Det)** — finds it after the fact |
| **Nature** | **Automated** — enforced by the application/database with no human judgement · **Manual** — performed by a person · **Hybrid (Auto+Manual)** — system-assisted but requires human review/sign-off |

Control IDs use the RCM taxonomy: `ITGC-AC/CM/SD/OP-nn` (IT general controls), `REV-nn`, `EXP-nn`, `INV-nn`, `GL-nn`, `REC-nn`, `TAX-nn`, `PAY-nn`, `CON-nn` (application controls), and `R01–R16` (SoD rules). The authoritative control population is `compliance/Invisible_ERP_SOX_RCM_v1.xlsx` (generated by `compliance/build_rcm.py`); the SoD ruleset is in `packages/shared/src/permissions.ts` (`SOD_RULES`).
